### Verified implementation evidence

- Immutable audit mutation rejection is exercised by `npm run db:verify`.
- Effective role, descendant organization scope, bounded delegation, source-revocation, and access-review revocation are exercised by `npm run access:verify`.
- Workflow publication, maker/checker denial, approval, optimistic versioning, audit, and outbox coupling are exercised by `npm run workflow:verify`.
- Access operating procedures are documented in `docs/runbooks/access-governance.md`.
- Document quarantine, scanner-computed integrity, MIME enforcement, promotion, signed download, audit, and outbox behavior are exercised by `npm run documents:verify`.
- Presigned upload, MinIO metadata, bucket promotion, and signed download are exercised against a real local MinIO service by `npm run object-storage:verify`.
- Document security operations are documented in `docs/runbooks/document-security.md`.
- Notification template publication, optimistic push consent, recipient-only inbox, read state, and opaque-payload structure are exercised by `npm run notifications:verify`.
- RabbitMQ publisher confirms, persistent event envelopes, PostgreSQL publication state, and retry scheduling are exercised by `npm run outbox:verify`.
- The web ERP shell provides OIDC/PKCE/nonce sign-in, encrypted HTTP-only BFF sessions, a live role-aware overview, scope-filtered approval inbox, request submission, decision actions, and the notification centre.
- Automated accessibility checks cover the shell landmark/navigation structure; production build and local sign-in preview are verified without public hosting.
- Search projections use strict OpenSearch mappings and bounded retry; API results are filtered in the index and re-authorized from PostgreSQL before disclosure.
- The API exposes bounded-label Prometheus process/request metrics plus separate liveness and PostgreSQL readiness probes; the on-premise topology keeps metrics and application/data tiers off published host ports.
- `npm run restore:verify` creates a checksum-protected logical backup and proves an isolated PostgreSQL recovery with migration history and a random control record. Production encryption, immutability, retention, RPO, and RTO remain explicit infrastructure/policy gates.
- `npm run load:verify` passes a 500-request, concurrency-25 runtime/PostgreSQL smoke baseline with a zero-error and p95 threshold; it is not substituted for the D-11 production capacity workload.
- GitHub quality gates reproduce Node 24 build, lint, tests, database migration and platform verifiers, restore proof, audit, and all four deployment-image builds.
- `npm run platform-slice:verify` correlates one clean document, independently approved request, recipient notification, authorized search result, forbidden search denial, audit trail, and outbox evidence under one request context using real PostgreSQL and OpenSearch.
- `npm run failure:verify` proves database-loss readiness isolation, continued API liveness, connection recovery, and stateless API process replacement against disposable CI services.
- `npm run contract:verify` boots the compiled API and validates the generated OpenAPI 3 document for required versioned operations, request schemas, bearer declarations, and absence of unversioned workflow routes.
- `npm run oidc:verify` serves an ephemeral JWKS and proves signature, issuer, audience, expiry, subject, session, and assurance-level handling, including negative tokens.

## Subsequent gates

Phases 2–7 follow the scope and gates defined in the architecture document. Each domain receives its own specification before implementation and may not bypass unresolved policy decisions with assumed rules.

Phase 2 implementation is governed by [PHASE_2_STUDENT_CORE_SPEC.md](phases/PHASE_2_STUDENT_CORE_SPEC.md). The specification keeps D-01/D-02/D-05/D-06/D-07/D-11-dependent rules unpublished while permitting synthetic, versioned foundations and verification work.

### Phase 2 implementation evidence

- The canonical student foundation creates only `PROVISIONAL` records, preserves immutable source provenance, enforces exact-retry idempotency, records append-only status history, couples audit/outbox evidence transactionally, and denies reads outside scope through `npm run students:verify`.
- Restricted alternate identifiers have no plaintext column: storage requires encrypted bytes plus a non-reversible lookup fingerprint. Identifier authority, matching, student-number format, and account-link policy remain blocked by D-01 rather than hardcoded.
- The curriculum regulation registry stores immutable versioned rule documents, requires scope and optimistic publication state, emits audit/outbox evidence, and defaults production publication off through `ACADEMIC_POLICY_PUBLICATION_ENABLED=false`; `npm run curriculum:verify` publishes only an explicitly empty synthetic preview.
- Registration periods and offerings are immutable after gated publication; requests are idempotent and remain pending until an enabled evaluator records its version, published regulation, trace, and reason. `npm run registration:verify` proves disabled gates, serialized capacity, append-only decisions, audit, and minimum-data events with synthetic rules only.
- Attendance stores one immutable raw observation per confirmed roster member and keeps raw presence separate from any institutional consequence. Finalization requires a complete roster and records a canonical evidence hash; approved corrections append maker-checker deltas instead of rewriting history. Both policy-dependent actions default off, while `npm run attendance:verify` proves scope isolation, gates, evidence immutability, correction separation, audit, and minimum-data events using synthetic data.
- The student subledger stores money only as integer minor units with ISO currency and creates balanced double-entry demand/payment/refund journals. Its provider-neutral boundary accepts only normalized, already-verified payment events with verifier/version/trace evidence; PostgreSQL and advisory locks prevent duplicate delivery or reconciliation snapshots that omit in-flight events. Immutable receipt manifests, scoped control-total reconciliation with maker-checker approval, partial-refund reservation/posting limits, and reversal/refund exclusion are implemented. All money-moving and approval gates default off until NIET rules and concrete providers are approved; `npm run finance:verify` proves scope, gates, concurrency, idempotency, balance-at-commit, immutability, reconciliation mismatch rejection, derived balance, audit, and minimum-data events.
- Finance accounts can originate against a non-terminal admission application under an explicit policy reference before student conversion. The original application owner is retained permanently; conversion appends an immutable account-to-student link in the same transaction, so pre-admission payments become visible in the canonical student record without rewriting ledger ownership. The gate defaults off until D-06/D-07 approval; `npm run admissions:verify` proves account replay, real payment continuity, post-conversion creation rejection, and atomic linkage.
- Fee structures are complete immutable versions containing dated installment lines, fee-head identifiers, integer amounts, and deterministic allocation order. Publication is maker-checker and policy-gated. Governed demands select published lines, post one balanced receivable journal, and retain per-line allocation evidence; a database constraint rejects missing, mismatched, cross-scope, or cross-currency allocations, while uniqueness prevents charging the same account for the same structure line twice. No NIET amounts, tax formulae, concessions, or scholarship rules are embedded. `npm run fee-structures:verify` proves concurrent replay, publication controls, allocation totals, duplicate prevention, direct-database bypass rejection, immutability, audit, and events.
- Migration batches retain only encrypted candidate payloads plus immutable source hashes, require exact row-count and canonical-hash reconciliation, and separate reconciliation from approval. Application defaults off and fails closed even when enabled unless a domain-specific approved adapter exists; `npm run migration:verify` proves scope, duplicate rejection, control totals, maker-checker, immutable evidence, audit, and minimum-data events without importing unverified records.
- Separate React Native CLI foundations now exist for NIET Student and NIET Staff with native Android/iOS projects, accessible role-specific preview shells, dark-mode support, no production mock records, and explicit fail-closed identity/offline/push posture. Workspace typecheck, lint, and render tests cover both apps; native signing, devices, and approved mobile-security policy remain acceptance gates documented in `docs/runbooks/mobile-build.md`.
- Admissions drafts retain encrypted application content plus integrity hashes, enforce exact idempotent retries and applicant-owned submission, freeze submitted payloads, and record evaluator/version/regulation evidence for every decision. Production decisions default off until D-06 is approved; `npm run admissions:verify` proves the encryption boundary, scope, ownership, gate, append-only evidence, audit, and minimum-data events.
- Timetable meetings use recurring weekday/minute slots with PostgreSQL exclusion constraints for room, instructor, and offering overlaps. Published meetings are immutable and publication defaults off until scheduling policy is approved; `npm run timetable:verify` proves scope, the gate, conflicting overlap rejection, adjacent-slot acceptance, audit, and minimum-data schedule events.
- Admission offers reference immutable terms manifests plus an explicit policy reference and configured expiry instant. They can be accepted or declined only by the applicant; late acceptance fails closed, while exceptional withdrawal/expiry separates the issuer from the actor and writes append-only lifecycle evidence. A scoped expiry worklist supports operations without inventing a default deadline. Accepted offers convert atomically through the student-domain service into one provenance-preserving `PROVISIONAL` student. Lifecycle and conversion gates default off until D-01/D-06 approval; `npm run admission-offers:verify` and `npm run admissions:verify` prove ownership, maker-checker, expiry, exact replay, atomic conversion, audit, and events.
- Accepted-but-unconverted offers support applicant-owned cancellation requests and versioned, explainable maker-checker assessment. A `NO_REFUND_REQUIRED` assessment can close the offer and application; `FINANCE_REVIEW_REQUIRED` blocks conversion and enters a scoped finance worklist. A separate third-party finance resolution can then cancel or reject the request using versioned policy evidence. `REFUND_COMPLETED` is accepted only when linked to an approved refund that has a real refund posting and an immutable evidence reference naming the same cancellation; admissions never fabricates or directly posts money. Converted admissions fail closed into the governed student-withdrawal path. The gate defaults off until D-06/D-07 approval; `npm run admission-cancellations:verify` proves ownership, replay, evidence immutability, governed finance closure, real-refund linkage enforcement, conversion exclusion, audit, and events.
- Seat matrices are immutable versions containing programme/cycle scope, category capacity, deterministic allocation order, and concrete reusable slot rows. Publication is maker-checker and database-verified against a canonical category manifest and exact slot counts. Reservation locks one available slot and records evaluator/version/policy/trace evidence; partial uniqueness prevents concurrent over-allocation. Governed offer issuance can require the reservation. Decline, withdrawal, expiry, and cancellation append release evidence and free capacity, while conversion appends consumption evidence and permanently occupies intake capacity. The gates default off until D-06 approval; `npm run seat-matrices:verify` proves concurrent exclusion, release/reuse, cancellation release, conversion consumption, availability derivation, immutability, audit, and events.
- Admission merit lists are immutable versions of externally evaluated ranking evidence. Each list retains the evaluator and version, policy and source-evidence references, programme/cycle scope, supplied display score, rank, deterministic allocation order, category, trace, and reason. PostgreSQL verifies the complete entry manifest during maker-checker publication. A separately gated reservation rule can require an exact published merit entry matching the application, programme, cycle, scope, and seat category; the ERP does not invent a merit formula or category policy. Both gates default off until D-06 approval; `npm run merit-lists:verify` proves concurrent replay, publication segregation, scope, immutable entries, deterministic reads, exact merit-to-seat linking, audit, and events.
- Admission conversion reconciliation scans canonical conversion records for five cross-domain invariants: converted application state, accepted offer state, admissions-owned student provenance, complete applicant-finance account linkage, and consumed seat evidence when a reservation exists. Findings are durable, fingerprinted, scope-filtered, and idempotent. A `RESOLVED` closure is rejected while the live invariant still fails; a policy waiver is separate, maker-checker controlled, and independently gated. Neither operation rewrites admission, student, finance, or seat evidence. Both closure gates default off until D-01/D-06/D-07 approval; `npm run conversion-exceptions:verify` proves legacy-defect detection, false-resolution rejection, repaired-link closure, waiver separation, immutable evidence, audit, and events.
- Canonical students can request governed withdrawal through a separate student-domain workflow. The scoped worklist exposes active holds, open registrations/programme enrolments, pending finance reviews, and non-zero account counts without converting those observations into invented policy. Approval is maker-checker and atomically closes confirmed/waitlisted registration plus provisional/active programme enrolment evidence before appending the canonical `WITHDRAWN` status history. Active/proposed holds, unresolved admission finance review, and undecided refunds fail closed. The gate defaults off until D-05/D-06/D-07 approval; `npm run student-withdrawals:verify` proves self ownership, blockers, dependency visibility, cross-domain closure, immutable history, audit, and events.
- `/api/v1/student-core/me` is a self-only, scope-checked read model over canonical student, confirmed registration, published timetable, finalized/effectively corrected attendance, and derived receivable balance. The web ERP exposes these live states without restricted identifiers or inferred attendance/finance policy; `npm run student-core:verify` proves the complete synthetic conversion-to-account journey and negative identity/scope access.
- Programme versions bind an immutable structure manifest to one published regulation. Student programme enrolments preserve named assignment-engine evidence and use PostgreSQL effective-date exclusion to prevent overlapping histories; activation is separately gated and maker-checker. `npm run programmes:verify` proves both gates, regulation/scope alignment, overlap rejection, immutability, audit, and minimum-data events.
- Student holds are explicit policy/evidence records and never inferred from balances or attendance. Only a separately activated `REGISTRATION_SUBMISSION` effect can block registration when enforcement is enabled; release appends immutable evidence. `npm run holds:verify` proves the disabled gate, maker-checker activation, opt-in enforcement, blocking/recovery, immutable release, audit, and events.
- Waitlisted requests create durable per-offering FIFO evidence. Promotion locks every requested offering, requires capacity and queue-head position for all items, separates the promoter from the original decision maker, and preserves promotion evaluator evidence. Applicant-owned withdrawal appends evidence and releases capacity; both gates default off. `npm run registration:verify` proves full-capacity rejection, withdrawal, promotion and concurrent seat serialization.
- Registration submission and add/drop windows are immutable, versioned configuration tied to a published academic period and organizational scope. Publication is maker-checker, PostgreSQL excludes overlapping published windows of the same type, and active-window lookup is scope protected. When enforcement is enabled, new registration requests require an open `SUBMISSION` window and confirmed-request withdrawal requires an open `ADD_DROP` window; waitlist withdrawal remains available so students can always release queue position. Exact idempotent retries remain valid after closure. Publication and enforcement gates default off until D-05 approval; `npm run registration-windows:verify` proves replay, overlap exclusion, scope, enforcement, post-close retry, immutability, audit, and events.
- Registration requests can retain one immutable request-time eligibility snapshot containing externally evaluated requested/maximum credit units, evaluator/version/policy/trace evidence, and an exact offering-manifest adviser approval when policy requires it. Adviser evidence is non-self, scope/period bound, idempotent, and immutable. The ERP independently derives conflicts from published meetings and the student's confirmed registrations; clients cannot supply the conflict result. Enforcement defaults off until D-05 approval and no NIET credit limit or adviser rule is embedded; `npm run registration-eligibility:verify` proves fail-closed enforcement, credit bounds, non-self approval, manifest matching, live/internal timetable conflicts, replay integrity, immutability, audit, and events.
- Registration exceptions use single-use maker-checker authorizations bound to the exact student, published period, offering manifest, exception type, evaluator/version/trace, policy, and source evidence. Submission consumes only an exact set of approved authorizations matching detected credit, adviser, or timetable exceptions; a separately approved capacity authorization is checked during confirmation. Waitlisted decisions can require immutable policy-supplied expiry terms, and the worker atomically cancels due requests, removes their waiting entries, records append-only expiration evidence, and emits audit/outbox events. No duration or exception rule is hardcoded, both production gates default off, and `npm run registration-overrides-expiry:verify` proves segregation, exact matching, single use, late-promotion denial, concurrent-safe idempotent expiry, and seat release.
